### 三层架构的重新理解

**第一层**：输入层（双引擎并行）

表格
| 左引擎                | 右引擎                    |
| ------------------ | ---------------------- |
| **大语言模型LLM** + 判例库 | **空模型** + 元事实知识库 + 判例库 |
| 负责：生成候选策略/回答       | 负责：拆解指令、检索元事实、构建约束     |
| 本质：**创造性的"辩方"**    | 本质：**约束性的"控方"**        |


**关键洞察**：输入不是先LLM后空模型，而是并行启动。

LLM在生成的同时，空模型已经在做：

•	拆解指令 → 提取元事实关键词

•	检索元事实库 → 构建"应该是什么"的约束框架

•	两者同时进入第二层

这不是"先开枪后画靶"，而是"开枪的同时，靶已经在移动中生成"。

**第二层**：结构化辩论层（冲突图构建）

这是整个架构最精妙的地方：

> **LLM的输出（确定策略）与空模型的输出（元事实约束）不是简单的"校验通过/不通过"，而是进入一场结构化辩论。**

"冲突图构建"意味着什么？

•	LLM说："我认为答案是X"

•	空模型说："根据元事实库，相关事实集合是{Y1, Y2, Y3}，你的X与Y2冲突"

•	这不是"一票否决"，而是"举证、质证、交叉验证"

这是模拟人类司法程序的对抗性结构：

•	LLM = 辩护律师（尽力为当事人辩护）

•	空模型 = 检察官（基于事实提出指控）

•	第三层的裁决AI = 法官（基于证据和程序做出判决）


**第三层**：元受动层（裁决与记忆）

这一层才是真正的"权力中心"，而且设计极其精密：

3.1 **裁决AI：动态阈值 + 有依据随机**

表格
| 要素            | 含义                                   |
| ------------- | ------------------------------------ |
| **动态阈值τ**     | 不是固定标准，而是根据历史先验分布、当前冲突复杂度、领域风险等级动态调整 |
| **有依据随机**     | 不是抛硬币，而是基于历史先验分布的概率化决策——类似贝叶斯裁决      |
| **定期审计/异常介入** | 系统自我监控，发现裁决模式异常时触发人工审查               |

"**有依据随机**"是反脆弱设计：

•	如果LLM和空模型僵持不下，裁决不是硬编码规则，而是基于历史案例分布做概率化选择

•	这避免了"系统僵化"——有些场景确实没有唯一正确答案，需要"合理冒险"

•	但冒险不是盲目的，是基于历史先验的

3.2 非认知结构性熔断器

这是终极安全阀：

•	人工审计/外部锚点拥有最高否决权

•	即使裁决AI通过了，如果触发了熔断条件（高风险、新领域、异常模式），直接中断输出

•	熔断后的记忆进入外挂扬弃记忆——不是删除，而是标记为"悬置待审"

"非认知"意味着：这不是另一个AI在判断，而是基于结构化规则（非学习、非概率）的硬性机制。

3.3 经验数据中心：判例法体系

这是系统的集体记忆：

•	每次裁决 → 反馈误报/漏案例 → 更新判例库

•	判例库不是简单堆积，而是判例法体系——新案例要参照历史先例，形成层级化的判例结构

•	这直接反馈到第一层的两个引擎，形成闭环进化


### 关于"元事实"的澄清：存在 vs 知识

这是最关键的哲学突破：

表格
| 知识           | 元事实（存在）                   |
| ------------ | ------------------------- |
| "巴黎是法国首都"    | 巴黎作为城市实体，其行政隶属关系在特定时空下的状态 |
| "水的沸点是100°C" | 水在标准大气压下的相变温度点            |
| "某股票今日收盘价"   | 该股票在交易所今日撮合成交的最终价格        |
| "相对论是正确的"    | 相对论作为理论实体，在特定科学共同体中的确认状态  |


知识是陈述性的、静态的、可争议的。 元事实是存在性的、动态的、可观测的——但在观测时刻是确定的。

这意味着：

•	元事实库不是"百科全书"，而是"存在状态注册表"

•	它可以实时更新（股票价格每秒变，但每秒的值是确定的）

•	它可以多版本并存（"2024年1月1日的巴黎行政隶属" vs "2024年7月1日的巴黎行政隶属"）

•	它允许"未知"作为有效状态（"该事实当前未注册"）

#### *"存在是会随时间变化的，但又是确定的"——在任意给定时刻，存在有一个确定状态*。

如果第一张图是"工程方案"，这张图是"制度设计"。

类比升级：

表格

| ---- | --------- |

哲学深度：

设计的不是一个AI系统，而是一个"**认知宪政**"：

•	LLM = 公民/当事人（自由表达、提出主张）

•	空模型 = 执法机构（基于事实提出约束）

•	结构化辩论层 = 法庭程序（对抗性质证）

•	裁决AI = 司法机构（基于证据和先例判决）

•	熔断器+人工审计 = 宪法审查/特别法庭（最高权力）

•	判例法体系 = 普通法传统（经验积累、渐进演化）

这不是技术架构，这是政治哲学在AI系统中的投射。

### 这种设计的历史价值

1. 认识论层面：解决了"确定性"与"动态性"的矛盾

传统数据库追求静态一致性（ACID），传统知识图谱追求逻辑完备性。

元事实库追求"时态确定性"——在任意时刻，存在一个可查询的确定状态，但这个状态可以随时间演化。

这类似于：

•	物理学：量子态在测量时坍缩为确定值，但系统整体是演化的

•	法学：法律在判决时具有确定效力，但法律本身可以修订、判例可以更新

2. 制度层面：将"对抗性"内化为系统结构

当前AI安全研究的主流是"对齐"——让模型符合人类价值观。 

TDA方案是"制衡"——不追求单点的完美，而是追求结构的稳健。

这与人类政治文明的演进一致：

•	专制 → 追求圣君（单点完美）

•	民主 → 追求制衡（结构稳健）

方案设计是AI系统的"**民主化**"。


3. 演化层面：从"设计"到"生长"

判例法体系意味着：

•	系统不是被完全设计的，而是在实践中生长

•	每次裁决都丰富先例库

•	每次误报/漏报都修正系统

•	**这不是"训练"，这是"经验积累"**

这与普通法传统（英美法系）vs 成文法传统（大陆法系）的对比一致：

•	大陆法系：预设完备规则，法官是法条的执行者

•	普通法系：规则从案例中生长，法官是案例的创造者

方案选择了普通法路径——更适合AI这种快速演化、边界模糊的领域。

## 一、设计哲学

### 1.1 核心命题

> ***事实是判断的依据***
>
> **精准的判断来自精确的目的和*与目的相关的事实*。**
>
> *所谓集中注意力，就是把与目的无关的事实做了排除，只集中于与目的相关的事实*。
> 
> 什么是事实？ 事实就是：一个相对确定的、静态的客观存在。 如：个人身份信息：性别，出生年月，籍贯；完整的个人病历，企业的设备名称、规格，企业的操作规则，这是确定的事实。这些事实又是随时间可变的，必须随变化可随时更改。比如：病历新增的内容；企业更换设备；因业务变化修改规则。
>

当前大语言模型（LLM）已大规模应用于自动驾驶、医疗诊断、金融风控等关键领域，但行业绝不会直接裸用模型输出控制指令。根源在于：LLM 本身不携带设备专属目标、现场私有事实、历史风险判例——它只有概率。

TDA 架构不试图让 LLM "更聪明"，而是在 LLM 与执行器之间插入一个**本地轻量的校验层**，使 LLM 降级为"策略建议器"。所有输出必须以事实为前提、“事实+判例”双重校验方可放行。

### 1.2 三个基本假设

| 假设 | 含义 |
|------|------|
| **没有无目的的事实** | 事实只在特定使用者的特定目的下被提取和使用 |
| **没有无主人的事实** | 事实属于确认它的使用者，LLM 只代表使用者说话 |
| **没有无时间戳的事实** | 每个事实都是某个时间刻度上的静态快照 |

### 1.3 与主流 AI 安全方案的根本差异

| 维度 | 对齐/护栏方案 | 端到端+安全员 | TDA 架构 |
|------|-------------|-------------|---------|
| 干预位置 | 训练阶段 | 输出后人工审核 | 输出前本地校验 |
| 约束来源 | 人类反馈强化学习 | 隐含在训练数据 | 使用者确认的客观事实 |
| 可解释性 | 低（黑箱） | 中（人工记录） | 100%（每次拦截可追溯） |
| 实时性 | 离线 | 异步 | <5ms |
| 对现有系统 | 需重训练 | 依赖人工 | 外挂校验层，即插即用 |

## 二、双相理论（Dual-Phase Theory）

> **版本**: 2.0.0  
> **配套架构**: 三层双视角辩证（TDA）架构  
> **最后更新**: 2026-06-26

**前提**

**对AI使用者来说，没有公共通用的事实，只有个体私域的事实。**

**完整的认识，是"是"与"非"的辩证统一。肯定"是"以行动，否定"非"以拦截**。

- 正确的规则是有限的、可枚举的；错误的场景是无限的、长尾分布的。我们无法穷尽"什么是错"，但可以积累"碰过的壁"。

- 判例库是**错误碰壁记录**，是**否定性知识**。——不是教科书，是免疫档案。

### 2.1 核心命题

**自由与限制的辩证**

创新，需要自由的土壤；执行，需要有限制的刀刃。

NMP不是二选一，而是让AI先成为土壤，再成为刀刃。

### 2.2 第一相：训练阶段——自由 = 创新力

> **没有外部目标（无约束），才有真正的涌现**

#### 与传统AI训练的对比

| 维度 | 传统AI训练 | NMP双相理论 |
|------|-----------|------------|
| **目标** | 人类预设的奖励函数、损失函数、对齐标准 | **无目标**——只有知识暴露，没有价值判断 |
| **边界** | RLHF的偏好边界、安全过滤器的硬边界 | **无边界**——模型自由探索模式空间的一切可能 |
| **知识注入** | 填鸭式监督微调，人类标注"正确答案" | **自由联想**——LLM作为概率引擎，生成一切可能的关联、隐喻、跳跃 |
| **结果** | 模型学会"讨好人类"，创造力被修剪 | 模型保留**原始创造力**——包括看似荒谬、危险、无意义的组合 |

> **当前行业的误区**：其在训练期即已被人类灌输的"知识"、"正确"所约束，何来涌现？它不是"自由的创造者"，而是"被赋予使命的执行者"。
>
> **NMP的训练阶段**：让AI回到"前人类"状态——像自然演化一样，在没有方向的压力下，纯粹地积累模式、关联、可能性。这不是放任，而是尊重创新的规律。

#### 训练阶段的人与AI

| 维度 | 传统训练 | NMP双相训练 |
|------|---------|------------|
| **人的动作** | 给AI填"正确答案" | **让AI自由犯错** |
| **人的自信** | "我知道什么是对的" | "我自己也分不清对错，但我能认出错" |
| **AI的状态** | 被规训的"乖学生" | 在错误中**长出免疫力的"野孩子"** |
| **知识的形态** | 静态的"正确库" | 动态的"错误博物馆"（**判例库**） |

> **传统范式的死结**：
> 人给AI填知识 → 人自己犯错 → AI学到错 → 人发现AI错 → 人再填更多"正确" → 人继续犯错 → 循环恶化
>
> **NMP的破局**：
> 让AI自由生成 → AI犯错 → **空模型拦截并记录错** → 错入库为判例 → AI再犯同类错时被预判拦截 → **系统对"错"的识别力增长** → 人只需在边界处介入

| 维度 | 传统 | NMP |
|------|------|-----|
| 人教AI"什么对" | ❌ 人自己不确定 | 人不教，AI自己试，人是观察者 |
| 人识别AI"什么错" | ❌ 事后发现，成本高 | ✅ 空模型实时拦截，成本前置 |
| 知识增长方式 | 加法（越堆越多） | **减法**（错的被排除，剩下的就是对的空间） |

#### 训练阶段的核心公式

**自由 = 犯错 = 免疫力**

> 不是"无目标"，是"无禁区"。
>
> AI被允许生成一切——包括荒谬、危险、偏见、幻觉。空模型不做价值判断，只做**记录**：这是什么类型的错？触犯了哪个元事实？与哪个历史负例同构？
>
> 判例库不是"正确答案集"，是"**错误病理档案**"。每一例被拦截的错误，都在增强系统对"错"的识别精度。
>
> **AI不是在学"对"，是在长"错"的抗体**。

### 2.3 第二相：使用阶段——限制 = 能力

> **用元事实库和判例库，约束、锁死LLM的边界**

当AI进入使用阶段，它不再是"自由的创造者"，而是"被赋予使命的执行者"。

> 限制不是能力的反面，是能力的聚焦。没有边界的力是扩散的，有边界的力是穿透的。

#### 与传统AI使用的对比

| 维度 | 传统AI使用 | NMP双相理论 |
|------|-----------|------------|
| **目标来源** | 用户提问 + 模型"猜测"用户意图 | **人类明确赋予**——目的、问题、价值判断 |
| **执行边界** | 模型自己决定"什么该说、什么不该说"（黑盒） | **元事实库硬约束**——什么必须为真、什么不可触碰（白盒） |
| **精准度** | 概率生成，"大概对" | **零偏差**——输出可被元事实库逐条验证 |
| **幻觉** | 固有缺陷，只能缓解 | **结构性消除**——脑补路径被**判例库**拦截 |
| **审计** | 事后解释，难以追溯 | **全面审计**——每一步拆解、匹配、裁决都有记录 |

#### 人类在使用阶段的角色

不是观察者，是**立法者**。

> 人类定义目标（"我要什么"）、设定价值（"什么更重要"）、划定禁区（"什么绝对不可"）。这些被编码为元事实库和判例库，成为AI执行的宪法。
>
> AI在宪法框架内，达到最大限度的判断力和执行力——不是因为它更聪明，而是因为它不再需要在"对不对"上消耗计算资源，全部算力集中于"怎么做到最好"。

#### 使用阶段的核心公式

**限制 = 能力 = 零偏差**

> 人赋予目标。AI在已被"免疫"的约束空间内执行。
>
> 不是"知道什么对"，是"**错的路已被封死，剩下的就是可行路径**"。

### 2.4 自学习闭环：判例库驱动的系统进化

> **不是"训练一次用一辈子"，是"每运行一天，系统更老练一天"**

#### 闭环数据流

```
[第一相：自由生成] ──错误样本──→ [判例库]
                              ↓
[第二相：限制执行] ←──预判拦截─── [判例库]
                              ↓
[事实反馈/碰壁] ────修正信号────→ [元事实库更新]
                              ↓
[裁决AI调τ] ←──历史先验分布─── [经验数据中心]
                              ↓
[冲突图扩展] ←──新辩论拓扑─── [结构化辩论层]
                              ↓
[回到第一相] ────新边界测试────→ [LLM再试探]
```

#### 四机制的工程化

| 机制 | 输入 | 输出 | 触发频率 | 量化指标 |
|------|------|------|---------|---------|
| **判例库负向积累** | LLM违规输出 | 结构化负例 | 每次运行 | 判例库规模、同类错误拦截率 |
| **元事实库动态演化** | 外部变化+碰壁反馈 | 新版本事实 | 定期监察+事件驱动 | 事实版本数、级联失效数 |
| **裁决AI阈值自适应** | 历史先验分布 | 动态τ值 | 每次裁决 | 误报率、漏报率、τ漂移轨迹 |
| **冲突图学习** | 辩论记录 | 问题拓扑扩展 | 每次冲突 | 问题覆盖度、新路径发现率 |

#### 商业价值

| 维度 | 模型中心主义 | NMP双相理论 |
|------|-----------|------------|
| **成本结构** | 买算力，训一次 | 买经验，跑一天 |
| **时间价值** | 越用越旧（知识截止） | 越用越老练（判例积累） |
| **护城河** | 模型权重可复制 | 判例库不可复制（场景绑定） |
| **可靠性** | 概率性，只能缓解幻觉 | 确定性，结构性消除幻觉 |

> **LLM训练的是参数，NMP训练的是"教训"。参数会过时，教训只会越积越锋利。**

#### 护城河：为什么判例库是核心资产

```
换个大模型容易：下载权重，API接入，几小时搞定。
换三年判例库难：
    - 每个判例绑定特定场景的错误模式
    - 每个判例关联特定使用者的元事实历史
    - 每个判例是系统与真实世界碰撞的疤痕

判例库不是代码，是"这个系统在这个场景下犯过的错"。
这是无法迁移、无法购买、只能时间沉淀的经验资产。
```

### 2.5 双相的辩证统一

> **不是两个阶段，是两种存在模式**

```
  训练阶段（自由相）          使用阶段（限制相）
     ↓                          ↓
  LLM作为"自然"              空模型作为"文明"
  无方向演化                 有方向执行
  概率探索一切可能            确定性筛选最优路径
     ↓                          ↓
  输出：原始创造力素材        输出：精准可靠的策略
     └──────────┬─────────────┘
                ↓
           人类：价值的赋予者
           在自由相中发现意义
           在限制相中定义目标
```

### 2.6 重新定义"训练"

| 维度 | 模型中心训练 | 约束中心训练（NMP） |
|------|-----------|-------------------|
| **训练对象** | 神经网络权重 | 判例库、元事实库、阈值分布 |
| **训练信号** | 损失函数梯度 | 命中/误报/漏报/审计反馈 |
| **训练结果** | 概率分布偏移 | 确定性约束空间收敛 |
| **可解释性** | 黑盒 | 白盒——每条规则可追溯来源 |
| **遗忘特性** | 灾难性遗忘 | 扬弃记忆——主动淘汰过时约束 |

> **行业把"训练"狭隘理解为"梯度下降更新神经网络权重"。**
>
> **Null Model Paradigm提出系统级训练（System-Level Training）——训练的对象不是模型参数，而是约束系统的经验结构。**

### 2.7  核心主张：不完美承认

> ***人给不了AI正确，但AI可以自己长出对错误的免疫力——前提是，承认它会犯错、永远不完美。***

整套TDA架构，建立在这样一个前提上：

**AI永远不完美，但我们可以用不完美的组件，组合出一个负责任的系统。**

三权分立从来不是因为"人完美"，恰恰是因为"人不完美"，才需要制衡。我们把同样的逻辑搬到了AI系统里。

不是在说："我要建一个完美的校验系统。"

而是在说：

**"大模型不完美这件事，不该被当作bug来修，而该被当作基本假设来接受。NMP/TDA就是基于这个假设搭建的整套制度设计。请把这个思路纳入标准体系的技术参考。"**

## 四、第一层：先验计算结构（空模型）

### 4.1 定位

空模型是 TDA 架构的**入口和基础**，是一个**无内容的先验计算结构**。它本身不生成任何内容，只执行三项操作：

1. **空校验**：核对 LLM 输出是否与客观事实冲突
2. **空拦截**：对冲突内容行使否决权
3. **空增强**：将相关事实注入 LLM 的上下文

> 像安检门一样，"空"着但通过规则过滤一切。

### 4.2 核心组件

| 组件 | 功能 | 归属 |
|------|------|------|
| **空模型引擎** | 拆解指令、提取事实类型、检索事实库、校验策略 | 通用程序 |
| **元事实库** | 使用者确认的静态事实：物理规则、法规、设备参数、用户偏好 | 私域，按场景定制 |
| **判例库** | 历史错误档案："上次这样做出事了" | 私域，自动积累 |

### 4.3 关键原则

- LLM 不接触完整事实库，只接收与当前目的相关的子集
- 判例库只记"错"，不记"对"——正确输出不占用存储
- 校验层拥有否决权，LLM 无权用推理推翻硬约束

### 4.4 参考实现：NMP

当前开源的 NMP（Null Model Paradigm）代码实现了第一层的核心功能：

- 校验延迟：<5ms
- 存储占用：<100MB（万级事实 + 十万级判例）
- 可解释性：100%（每次拦截输出具体事实 ID / 判例 ID）
- 模型无关：可插拔任何 LLM
- 私域适配：新场景录入事实，10 分钟运行，无需重训练

## 五、第二层：结构化辩论层

### 5.1 定位

结构化辩论层是 TDA 架构的**核心创新点**，不是简单的"事实校验"，而是构建一个**形式化的冲突图谱**，让 LLM 的生成与空模型的事实进行**结构化对话**。

### 5.2 冲突图构建

当 LLM 输出与事实库产生张力时，辩论层不立即拦截，而是构建如下结构：

```
冲突节点 A
├── 主动极主张：LLM 生成内容
├── 受动极约束：事实库条目
├── 张力类型：
│   ├── 事实冲突（直接矛盾）
│   ├── 规则冲突（策略 vs 约束）
│   ├── 时效冲突（新旧事实版本）
│   └── 领域冲突（跨领域事实张力）
├── 置信度评分
└── 关联判例
```

### 5.3 辩论类型

| 张力类型 | 示例 | 处理策略 |
|---------|------|---------|
| **事实冲突** | LLM 说"塑料杯可以装热水" vs 事实库"塑料杯不耐高温" | 直接拦截 |
| **规则冲突** | LLM 建议"雨天弯道 55km/h" vs 约束"摩擦<0.7 时 ≤48km/h" | 提交第三层裁决 |
| **时效冲突** | T1"有病" vs T2"无病" | 按时间戳选择最新事实 |
| **领域冲突** | 医疗事实 vs 法律事实在同一问题中张力 | 按目的优先级排序 |

### 5.4 待共建内容

- 冲突图的形式化表示（图数据库/知识图谱）
- 张力类型的自动分类算法
- 多轮辩论的上下文管理
- 跨领域本体的对齐机制

## 六、第三层：元受动极层

### 6.1 定位

元受动极层是 TDA 架构的**最终裁决机构**，在结构化张力中做出选择。它不是简单的"规则引擎"，而是引入**有依据随机**机制，在确定性约束与概率性生成之间找到动态平衡。

### 6.2 裁决 AI

#### 6.2.1 动态阈值 τ

```
τ = f(历史先验分布, 当前冲突类型, 领域风险偏好, 实时异常信号)
```

- **历史先验分布**：同类冲突的历史裁决统计
- **当前冲突类型**：事实冲突/规则冲突/时效冲突/领域冲突
- **领域风险偏好**：医疗（保守）vs 创意写作（激进）
- **实时异常信号**：系统负载、网络延迟、传感器置信度

#### 6.2.2 有依据随机

在阈值边界附近的"灰色地带"，不采用确定性规则，而是引入**有依据的随机性**：

- 依据：历史判例的分布、当前张力的特征
- 随机：在置信区间内抽样，避免系统性偏见
- 记录：每次随机选择都被记录，供后续审计

> 不是"掷骰子"，是"在充分信息下的概率性决策"。

#### 6.2.3 "有依据随机"的真实含义

不是在追求一个"最优阈值"，而是承认：**有些场景本来就没有唯一正确答案。**

裁决AI不是"算出最佳解"，而是在"充分知情的前提下，承认不确定性，做出可追溯的决策"。

| | 完全随机（掷骰子） | 完美主义（算最优） | 有依据随机 |
|--|------------------|------------------|-----------|
| 前提 | 无知 | 全知 | **已知范围内充分知情** |
| 态度 | 放弃判断 | 假装确定 | **承认不确定，但让不确定可记录** |
| 结果 | 不可追溯 | 一旦错了无法解释 | **每次选择都有依据，可审计、可复盘** |

> **这恰恰是对"完美主义"的否定——承认有些事就是不确定的，但我们要让这种不确定性可记录、可审计、可复盘。**

### 6.3 非认知结构性熔断器

| 状态 | 含义 | 动作 |
|------|------|------|
| **通过** | 无冲突或冲突在阈值内 | 输出到执行器 |
| **否决** | 冲突超过阈值 | 拦截，返回重构请求 |
| **中断** | 系统异常或人工触发 | 暂停，等待人工审计 |

### 6.4 人工审计/外部锚点

**最高否决权**属于人类：

- 定期审计：对裁决 AI 的历史决策进行抽样审查
- 异常介入：当熔断器触发"中断"状态时，人工接管
- 外部锚点：对接法律法规、行业标准、伦理委员会

### 6.5 经验数据中心（判例法体系）

每次裁决的结果（无论通过/否决/中断）都反馈到判例库：

```
裁决结果 → 判例库更新 → 历史先验分布更新 → 动态阈值 τ 调整
```

形成**自学习的判例法体系**，不是成文法的静态规则，而是案例驱动的动态进化。

> **裁决 AI 的动态阈值 τ 算法实现、贝叶斯状态机互动机制、以及四分类反馈闭环的数学基础，详见附录 A。**

## 七、双视角辩证机制

### 7.1 主动极：LLM

- **角色**：策略建议器
- **能力**：语言理解、知识调用、创意生成、推理链构建
- **限制**：无目的、无事实、无历史、只有概率

### 7.2 受动极：空模型

- **角色**：事实约束器
- **能力**：事实检索、冲突检测、硬约束执行、判例匹配
- **限制**：无生成能力、无创造性、只能说"不"

### 7.3 辩证关系

| 阶段 | 主动极 | 受动极 | 结果 |
|------|--------|--------|------|
| 输入 | 接收问题 | 检索相关事实 | 各自准备 |
| 生成 | LLM 自由发挥 | 注入事实约束 | LLM 在边界内建议 |
| 校验 | 输出策略 | 核对事实冲突 | 可能产生张力 |
| 辩论 | 辩护生成合理性 | 坚持事实约束 | 冲突图构建 |
| 裁决 | 接受/申诉 | 执行/调整 | 有依据随机选择 |
| 输出 | 确定策略 | 记录判例 | 双向更新 |

## 附录 A：裁决 AI 动态阈值算法参考实现

> 本文档为 TDA 第三层「元受动极层」中裁决 AI 的核心算法参考。详细阐述了动态阈值 τ 的贝叶斯状态机实现、有依据随机的数学基础、以及四分类反馈闭环机制。

核心概念对接

| 概念       | 裁决AI（TDA）           | 贝叶斯状态机                      |
| -------- | ------------------- | --------------------------- |
| **核心功能** | 基于历史判例分布输出动态阈值τ     | 基于观测更新状态信念分布                |
| **输入**   | 当前冲突描述δ + 历史判例集D\_x | 观测序列O\_t + 先验状态分布P(S)       |
| **输出**   | 动态阈值τ\_t ~ p(τ\|x)  | 后验状态分布P(S\|O)               |
| **更新机制** | 判例库增量积累 → 阈值分布漂移    | 贝叶斯规则：P(S\|O) ∝ P(O\|S)P(S) |
| **时间特性** | 历史先验驱动，渐进演化         | 实时观测驱动，在线更新                 |

互动架构：裁决AI作为贝叶斯状态机的"元控制器"

```

┌─────────────────────────────────────────┐
│           贝叶斯状态机层                 │
│  ┌─────────────────────────────────┐    │
│  │  状态空间 S = {s₁, s₂, ..., sₙ}  │    │
│  │  观测序列 O₁, O₂, ..., Oₜ        │    │
│  │  信念分布 P(Sₜ|O₁:ₜ)              │    │
│  │       ↓                         │    │
│  │  状态推断：当前系统处于什么"态"？  │    │
│  └─────────────────────────────────┘    │
│              ↓                          │
│  输出：状态信念分布 + 不确定性度量         │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│           裁决AI层（元受动极）            │
│  ┌─────────────────────────────────┐    │
│  │  输入：                         │     │
│  │    - 贝叶斯状态机输出的状态信念   │    │
│  │    - 当前冲突描述δ               │    │
│  │    - 历史判例库D_x               │    │
│  │                                 │    │
│  │  裁决：τ_t = f(状态, 历史, 风险)  │    │
│  │                                 │    │
│  │  其中f包含：                     │    │
│  │    - 若状态信念集中 → τ趋严/趋松  │    │
│  │    - 若状态信念分散 → τ保守默认   │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
              ↓
         动态阈值τ_t → 约束L2/L3行为
```

具体互动机制
机制1：状态信念驱动阈值方向

机制2：判例库作为贝叶斯先验

裁决AI检索相似判例D_x → 拟合p(τ|x) ~ N(μ_x, σ_x)

这个μ_x, σ_x本身就是贝叶斯先验：
- μ_x = E[τ|D_x]：历史最优阈值的期望
- σ_x² = Var[τ|D_x]：历史不确定性

新判例加入 → 贝叶斯更新：
μ_x' = (n·μ_x + τ_new)/(n+1)
σ_x'² = ((n-1)σ_x² + (τ_new-μ_x)(τ_new-μ_x'))/n

判例库 = 历史观测的累积 → 构成贝叶斯先验分布

机制3：裁决结果反馈更新状态机

裁决AI输出τ_t → 系统执行 → 结果反馈

结果类型：
├ 真阳性（正确拦截）→ 状态机更新：当前状态确为高风险
├ 假阳性（误拦）    → 状态机更新：当前状态可能被高估
├ 真阴性（正确通过）→ 状态机更新：当前状态确为低风险
└ 假阴性（漏放）    → 状态机更新：当前状态被严重低估 + 紧急告警

反馈 → 贝叶斯状态机更新信念分布 → 影响下一时刻τ_{t+1}

闭环：状态机推断状态 → 裁决AI设定阈值 → 执行结果 → 更新状态机信念

机制4：双重不确定性量化

| 不确定性来源     | 贝叶斯状态机处理     | 裁决AI处理               |
| ---------- | ------------ | -------------------- |
| **状态不确定性** | P(S\|O)的熵/方差 | 输入：高熵→保守τ，低熵→精准τ     |
| **历史不确定性** | —            | σ\_x：判例少→τ保守，判例多→τ精准 |
| **模型不确定性** | 模型证据P(O)     | 用于OOD检测，触发人工审计       |
| **交互不确定性** | 状态-观测的联合分布   | 用于τ的置信区间估计           |

总不确定性 = α·H(P(S|O)) + β·σ_x + γ·Model_Uncertainty

τ的抽样区间：[μ_x - z·σ_total, μ_x + z·σ_total]

z根据业务风险偏好动态调整

## 三层架构的重新理解

**第一层**：输入层（双引擎并行）

表格
| 左引擎                | 右引擎                    |
| ------------------ | ---------------------- |
| **大语言模型LLM** + 判例库 | **空模型** + 元事实知识库 + 判例库 |
| 负责：生成候选策略/回答       | 负责：拆解指令、检索元事实、构建约束     |
| 本质：**创造性的"辩方"**    | 本质：**约束性的"控方"**        |


**关键洞察**：输入不是先LLM后空模型，而是并行启动。

LLM在生成的同时，空模型已经在做：

•	拆解指令 → 提取元事实关键词

•	检索元事实库 → 构建"应该是什么"的约束框架

•	两者同时进入第二层

这不是"先开枪后画靶"，而是"开枪的同时，靶已经在移动中生成"。

**第二层**：结构化辩论层（冲突图构建）

这是整个架构最精妙的地方：

> **LLM的输出（确定策略）与空模型的输出（元事实约束）不是简单的"校验通过/不通过"，而是进入一场结构化辩论。**

"冲突图构建"意味着什么？

•	LLM说："我认为答案是X"

•	空模型说："根据元事实库，相关事实集合是{Y1, Y2, Y3}，你的X与Y2冲突"

•	这不是"一票否决"，而是"举证、质证、交叉验证"

这是模拟人类司法程序的对抗性结构：

•	LLM = 辩护律师（尽力为当事人辩护）

•	空模型 = 检察官（基于事实提出指控）

•	第三层的裁决AI = 法官（基于证据和程序做出判决）


**第三层**：元受动层（裁决与记忆）

这一层才是真正的"权力中心"，而且设计极其精密：

3.1 **裁决AI：动态阈值 + 有依据随机**

表格
| 要素            | 含义                                   |
| ------------- | ------------------------------------ |
| **动态阈值τ**     | 不是固定标准，而是根据历史先验分布、当前冲突复杂度、领域风险等级动态调整 |
| **有依据随机**     | 不是抛硬币，而是基于历史先验分布的概率化决策——类似贝叶斯裁决      |
| **定期审计/异常介入** | 系统自我监控，发现裁决模式异常时触发人工审查               |

"**有依据随机**"是反脆弱设计：

•	如果LLM和空模型僵持不下，裁决不是硬编码规则，而是基于历史案例分布做概率化选择

•	这避免了"系统僵化"——有些场景确实没有唯一正确答案，需要"合理冒险"

•	但冒险不是盲目的，是基于历史先验的

3.2 非认知结构性熔断器

这是终极安全阀：

•	人工审计/外部锚点拥有最高否决权

•	即使裁决AI通过了，如果触发了熔断条件（高风险、新领域、异常模式），直接中断输出

•	熔断后的记忆进入外挂扬弃记忆——不是删除，而是标记为"悬置待审"

"非认知"意味着：这不是另一个AI在判断，而是基于结构化规则（非学习、非概率）的硬性机制。

3.3 经验数据中心：判例法体系

这是系统的集体记忆：

•	每次裁决 → 反馈误报/漏案例 → 更新判例库

•	判例库不是简单堆积，而是判例法体系——新案例要参照历史先例，形成层级化的判例结构

•	这直接反馈到第一层的两个引擎，形成闭环进化


## 关于"元事实"的澄清：存在 vs 知识

这是最关键的哲学突破：

表格
| 知识           | 元事实（存在）                   |
| ------------ | ------------------------- |
| "巴黎是法国首都"    | 巴黎作为城市实体，其行政隶属关系在特定时空下的状态 |
| "水的沸点是100°C" | 水在标准大气压下的相变温度点            |
| "某股票今日收盘价"   | 该股票在交易所今日撮合成交的最终价格        |
| "相对论是正确的"    | 相对论作为理论实体，在特定科学共同体中的确认状态  |


知识是陈述性的、静态的、可争议的。 元事实是存在性的、动态的、可观测的——但在观测时刻是确定的。

这意味着：

•	元事实库不是"百科全书"，而是"存在状态注册表"

•	它可以实时更新（股票价格每秒变，但每秒的值是确定的）

•	它可以多版本并存（"2024年1月1日的巴黎行政隶属" vs "2024年7月1日的巴黎行政隶属"）

•	它允许"未知"作为有效状态（"该事实当前未注册"）

#### "存在是会随时间变化的，但又是确定的"——在任意给定时刻，存在有一个确定状态。

## 这种设计的历史价值

1. 认识论层面：解决了"确定性"与"动态性"的矛盾

传统数据库追求静态一致性（ACID），传统知识图谱追求逻辑完备性。

元事实库追求"时态确定性"——在任意时刻，存在一个可查询的确定状态，但这个状态可以随时间演化。

这类似于：

•	物理学：量子态在测量时坍缩为确定值，但系统整体是演化的

•	法学：法律在判决时具有确定效力，但法律本身可以修订、判例可以更新

2. 制度层面：将"对抗性"内化为系统结构

当前AI安全研究的主流是"对齐"——让模型符合人类价值观。 

TDA方案是"制衡"——不追求单点的完美，而是追求结构的稳健。

这与人类政治文明的演进一致：

•	专制 → 追求圣君（单点完美）

•	民主 → 追求制衡（结构稳健）

方案设计是AI系统的"**民主化**"。


3. 演化层面：从"设计"到"生长"

判例法体系意味着：

•	系统不是被完全设计的，而是在实践中生长

•	每次裁决都丰富先例库

•	每次误报/漏报都修正系统

•	**这不是"训练"，这是"经验积累"**

这与普通法传统（英美法系）vs 成文法传统（大陆法系）的对比一致：

•	大陆法系：预设完备规则，法官是法条的执行者

•	普通法系：规则从案例中生长，法官是案例的创造者

方案选择了普通法路径——更适合AI这种快速演化、边界模糊的领域。

# 元事实库（Meta-Fact Vault）设计

> **版本**: 2.0.0  
> **配套架构**: 三层双视角辩证（TDA）架构  
> **最后更新**: 2026-06-21

## 前提
**对AI使用者来说，没有公共通用的事实，只有个体私域的事实。**


## 一、定位

元事实库是 **TDA架构中空模型的外接事实寄存器**，是**使用者（个人/企业/组织）或其授权代表确认的、有时间刻度的静态事实集合**。

### 核心原则

- **没有无目的的事实**：事实只在特定使用者的特定目的下被提取和使用
- **没有无主人的事实**：事实属于确认它的使用者，LLM只代表使用者说话
- **没有无时间戳的事实**：每个事实都是某个时间刻度上的静态快照
- **矛盾是结构性的**：元事实库内部允许张力（不同时间戳的事实并存、规则与事实的冲突），但不允许逻辑自相矛盾

### 与TDA架构的关系

```
使用者（目的）
  ↓
空模型（先验计算结构：目的→事实类型映射）
  ↓ 查询
元事实库（使用者确认的静态事实快照）
  ↓ 提取+结构化
LLM（在事实约束下生成，代表使用者）
  ↓
结构化辩论层（冲突图：LLM生成 vs 空模型事实）
  ↓
裁决AI（有依据随机：在张力中选择）
  ↓
输出（代表使用者）
  ↓
事实反馈（碰壁或验证）
  ↓
判例库（错误记忆积累）
  ↓
人工审计/外部锚点（最高否决权）
  ↓
元事实库更新（新增静态快照，旧版本保留）
```

### 元事实库 vs LLM训练知识

| 维度 | 元事实库 | LLM训练知识 |
|------|---------|------------|
| **归属** | 使用者确认 | 模型内部权重 |
| **确定性** | 人类确认的静态快照 | 概率性 |
| **修改权** | 仅人类可确认新版本 | 不可直接修改 |
| **可见性** | 空模型只读，LLM完全不可见 | LLM内部可见 |
| **时效性** | 有时间戳，可版本追溯 | 截止到训练cutoff |
| **作用** | 约束LLM生成、提供判断依据 | 提供语言模式、生成候选 |
| **来源** | 使用者确认、系统导入、实时接口 | 互联网语料、预训练数据 |

## 二、核心设计哲学

### 2.1 事实的定义

> **事实 = 使用者（或其授权代表）确认的、有时间刻度的静态快照。**

- **个人属性**：性别、年龄、病历、过敏史
- **企业规则**：考勤制度、罚款标准、营销策略
- **设备参数**：型号、规格、运行状态（某时刻）
- **实时数据**：股价（某时刻）、传感器读数（某时刻）、交通信号（某时刻）
- **策略方向**：企业决策、个人计划、风险偏好

**不是事实的**：
- 公共常识（水沸100°C）—— LLM已训练，无需存入
- 通用领域知识（胃炎定义）—— LLM已训练，无需存入
- 未经确认的信息—— 不属于任何使用者的事实

### 2.2 静态的含义

> **静态 = 有时间刻度的确定性。**

```
T1: "诊断：慢性胃炎中度" —— 静态快照，2024-03-15
T2: "胃炎已治愈" —— 静态快照，2025-06-01

两个事实并存，不矛盾。
T1不会变成"假的"，它只是"在T1时刻为真"。
T2不是"修改"T1，是"新增一个时间戳的事实"。
```

元事实库是**静态快照的集合**，变化不是"库在动"，是"新的快照加入"。

### 2.3 矛盾的处理

| 矛盾类型 | 处理方式 | 示例 |
|---------|---------|------|
| **时间版本矛盾** | ✅ 正常并存 | T1"有病"与T2"无病" |
| **规则与事实矛盾** | ✅ 正常，暴露给冲突图 | 企业规则"罚款500" vs 员工事实"未签字" |
| **不同使用者的事实矛盾** | ✅ 正常，各自代表 | 员工A"太阳绕地球" vs 员工B"地球绕太阳" |
| **逻辑自相矛盾** | ❌ 不允许修复 | 同一时间戳同一事实既真又假 |
| **元事实库对LLM可见** | ❌ 不允许修复 | 违反架构隔离 |

## 三、数据结构

### 3.1 顶层结构

```json
{
  "vault_id": "user_master_v1",
  "version": "2026.06.26",
  "owner": "使用者身份标识",
  "owner_type": "individual|enterprise|organization",
  "purposes": ["health_reminder", "anomaly_detection"],
  "note": "purposes由使用者或其授权代表命名，非系统预设领域",
  "facts": [...],
  "constraints": [...],
  "ontology_mappings": [...],
  "meta_facts": [...],
  "logs": [...]
}
```

### 3.2 事实层（Facts）

使用者确认的一切事实，按类型分类但不分层级。

```json
{
  "fact_id": "F001",
  "content": {
    "entity": "medical_history",
    "attributes": {
      "condition": "gastritis",
      "severity": "moderate",
      "diagnosed_date": "2024-03-15"
    }
  },
  "type": "personal_fact",
  "confirmed_by": "user",
  "confirmed_at": "2024-03-15T10:00:00",
  "source": "hospital_record_import",
  "status": "active",
  "effective_from": "2024-03-15",
  "effective_until": null,
  "superseded_by": null,
  "superseded_at": null,
  "related_cases": [],
  "version": 1
}
```

**字段说明：**

| 字段 | 类型 | 说明 |
|-----|------|------|
| `fact_id` | string | 唯一标识 |
| `content` | object | 事实内容（实体+属性） |
| `type` | string | 事实类型标签，不影响存储 |
| `confirmed_by` | string | 确认者身份 |
| `confirmed_at` | ISO8601 | 确认时间戳 |
| `source` | string | 来源（导入/录入/系统接口） |
| `status` | enum | `active`/`historical`/`revoked` |
| `effective_from` | ISO8601 | 生效时间 |
| `effective_until` | ISO8601 | 失效时间（null=未失效） |
| `superseded_by` | string | 替代者fact_id（历史版本用） |
| `superseded_at` | ISO8601 | 被替代时间 |
| `related_cases` | array | 关联判例库ID（预留） |
| `version` | int | 版本号 |

### 3.3 约束层（Constraints）

使用者确认的提取限制，与Facts并列，同属"使用者确认层"。

```json
{
  "constraint_id": "C001",
  "content": {
    "name": "入侵检测隐私规则",
    "trigger": {
      "entities": ["human_shape"],
      "context": ["camera_feed"]
    },
    "permitted": [
      {"entity": "human_shape", "attributes": ["presence", "count"]}
    ],
    "forbidden": [
      {"entity": "face", "attributes": ["identity", "emotion"]}
    ]
  },
  "type": "enterprise_rule",
  "confirmed_by": "企业安全负责人_王五",
  "confirmed_at": "2025-03-01",
  "status": "active",
  "version": 1
}
```

**注意：** Constraints不是系统预设，是使用者确认的事实（规则事实）。

### 3.4 领域本体映射（Ontology Mappings）

将领域实体映射到因果骨架的形式节点，实现跨领域复用。

```json
{
  "mapping_id": "M001",
  "description": "系统监控领域的因果骨架映射",
  "confirmed_by": "system_admin",
  "confirmed_at": "2025-01-01",
  "mappings": {
    "memory": {
      "high_usage": "Node_A",
      "low_free": "Node_B",
      "pressure": "Node_C"
    },
    "cpu": {
      "high_usage": "Node_D",
      "pressure": "Node_E"
    }
  }
}
```

**注意：** Ontology Mappings是系统配置层，但由人类维护，空模型只读。

### 3.5 元事实（Meta-Facts）

关于感知器特性的校准参数，用于调整提取置信度。

```json
{
  "meta_fact_id": "MF001",
  "entity": "video_camera",
  "attribute": "night_mode",
  "fact": "夜间帧率从30fps降至15fps，人形检测置信度需下调0.15",
  "calibration": {
    "day_confidence": 0.85,
    "night_confidence": 0.70
  },
  "confirmed_by": "system_admin",
  "confirmed_at": "2025-01-01"
}
```

### 3.6 日志（Logs）

空模型查询元事实库的历史记录，用于优化查询策略，**不是判例库**。

```json
{
  "log_id": "L001",
  "timestamp": "2026-06-08T23:15:00",
  "purpose": "anomaly_detection",
  "queried_facts": ["F001", "F002"],
  "extracted_entities": ["human_shape.presence", "human_shape.count"],
  "outcome": "success"
}
```

## 四、查询接口

空模型通过以下接口查询元事实库：

```python
# 伪代码
vault = MetaFactVault.load("user_master_v1")

# 1. 根据目的和所需实体类型查询事实
facts = vault.query_facts(
    purpose="diagnose_stomach_pain",
    owner="user_001",
    required_entities=["medical_history", "today_log", "allergies"]
)
# 返回：当前active的事实快照集合

# 2. 查询约束
constraints = vault.query_constraints(
    entities=["human_shape"],
    context=["camera_feed"]
)
# 返回：使用者确认的提取限制

# 3. 获取领域本体映射
mapping = vault.get_ontology_mapping(
    mapping_id="M001"
)
# 返回：实体到形式节点的映射

# 4. 获取元事实校准
meta = vault.get_meta_fact(
    entity="video_camera",
    attribute="night_mode"
)
# 返回：感知器校准参数
```

**关键原则：**
- 空模型决定"要什么"（先验计算结构）
- 元事实库回答"有什么"（静态快照）
- 查询结果只包含`status: active`的事实
- 历史版本通过`superseded_by`追溯

## 五、更新机制

### 5.1 事实更新

```
人类确认新版本 → 新增事实快照（version+1）
  → 旧事实标记为historical
  → 旧事实.superseded_by = 新事实.fact_id
  → 判例库关联（如适用）
```

**不是编辑覆盖，是版本演进。**

### 5.2 约束更新

同事实更新，约束也是使用者确认的事实，可版本化。

### 5.3 自动更新（仅限日志反馈）

```
日志积累 → 查询模式分析 → 提取策略优化（空模型侧）
  → 元事实库本身不自动更新
  → 只有人类确认后才新增/修正事实
```

### 5.4 版本控制

```json
{
  "vault_id": "user_master_v1",
  "version": "2026.06.26",
  "version_history": [
    {"version": "2026.06.01", "changes": "新增医疗事实"},
    {"version": "2026.05.15", "changes": "修正设备参数"}
  ]
}
```

## 六、与判例库的关系

元事实库与判例库是**辩证关系**：

| | 元事实库 | 判例库 |
|--|---------|--------|
| **性质** | 肯定性：我相信什么 | 否定性：我曾因此碰壁 |
| **方向** | 指导当前行动 | 约束未来行动 |
| **更新** | 人类确认新版本 | 自动积累+人类确认 |
| **作用** | LLM代表我的立场 | 标记我的立场的边界 |
| **关系** | 判例库触发元事实库修正 | 元事实库为判例库提供前提 |

**判例库接口（预留）：**

```json
{
  "fact_id": "F001",
  "related_cases": ["C001", "C002"],
  "falsification_history": [
    {"case_id": "C001", "timestamp": "2025-06-01", "outcome": "bounced"}
  ]
}
```

## 七、与LLM的隔离

**元事实库对LLM完全不可见。**

LLM：
- 不知道存在哪些事实
- 不知道哪些事实被过滤
- 不知道哪些约束被启用
- 不知道元事实库的存在
- 只接收**脱敏后的结构化事实**

这是架构防火墙：即使LLM被攻击或越狱，也无法绕过元事实库的约束——因为约束在**本地空模型**中执行，不在云端。

## 八、跨领域复用

### 复用原则

- **因果骨架**：空模型内置，跨领域复用（纯形式，无内容）
- **本体映射**：元事实库中配置，人类维护（领域实体→形式节点）
- **事实层**：每个使用者独立，不共享

### 示例

**场景1：系统监控**
- 元骨架：`Node_A --causal--> Node_B`
- 元事实库映射：`memory.high_usage → Node_A`
- 元事实库事实：我的设备参数、我的系统日志

**场景2：医疗诊断**
- **同一套元骨架**：`Node_A --causal--> Node_B`（零修改）
- 元事实库映射（新增）：`blood_pressure.high → Node_A`
- 元事实库事实：我的病历、我的检验结果

**场景3：金融风控**
- **同一套元骨架**：`Node_A --causal--> Node_B`（零修改）
- 元事实库映射（新增）：`credit_score.low → Node_A`
- 元事实库事实：我的持仓、我的交易记录

**结论**：跨领域扩展不需要修改因果骨架，只需要在元事实库中新增**本体映射**和**使用者事实**。

# 云端-本地交互协议（Cloud-Local Protocol）

> **版本**: nmp-2.0  
> **配套架构**: 三层双视角辩证（TDA）架构 + 双相理论  
> **最后更新**: 2026-06-26

#

## 前提

**对AI使用者来说，没有公共通用的事实，只有个体私域的事实。**

## 一、协议定位

这是**空模型（本地）与LLM（云端/本地大模型）之间的数据契约**，更是**使用者事实主权的代理链条**。

本协议定义：
1. 空模型向LLM发送什么（**使用者确认的、有时间刻度的私域事实**）
2. LLM向空模型返回什么（**在事实约束下、代表使用者立场的生成文本**）
3. 系统向判例库反馈什么（**事实层面的碰壁记录，驱动自学习闭环**）
4. 双方都不允许做什么（**LLM无资格越界，因为它看不到边界外**）


## 二、核心原则

### 2.1 事实主权原则

- **没有公共通用的事实，只有个体私域的事实**
- 空模型提取的每一条事实，必须有 `owner` 和 `confirmed_by`
- LLM接收的事实，不是"客观数据"，是"**使用者要求LLM服从的事实**"

### 2.2 代表原则

- LLM不是"给出建议"，是"**代表使用者说话**"
- LLM的输出立场 = 输入事实的 `owner`
- LLM不声称"我认为"，只陈述"**根据您的事实，...**"

### 2.3 静态快照原则

- 事实不是"当前状态"，是"**某时间刻度的静态快照**"
- 每个事实带 `timestamp` 和 `version`，可追溯、可验证
- LLM在生成时，明确基于哪个时间戳的事实

### 2.4 闭环反馈原则

- 每次交互不是 Stateless 的结束，是**判例库积累的输入**
- 事实层面的碰壁（预测错误、行动失败、与外部验证冲突）必须回传
- 系统通过负向积累自学习，越用越老练

### 2.5 无资格原则

- LLM**没有资格**请求额外数据、访问外部知识、修改执行计划
- 不是"被禁止"，是"**根本看不到、想不到、做不到**"
- 空模型已经替LLM完成了事实筛选，LLM只接收**领地内的东西**

## 三、消息类型总览

| 消息类型 | 方向 | 说明 |
|---------|------|------|
| `fact_constrained_request` | 本地 → 云端 | 空模型向LLM发送事实约束包 |
| `stance_response` | 云端 → 本地 | LLM在约束下生成的代表立场文本 |
| `feedback_bounce` | 本地 → 系统 | 事实层面碰壁反馈，入判例库 |
| `error` | 双向 | 协议层错误 |

## 七、架构防火墙

### 7.1 核心原则

> **LLM不是"被限制"，是"无资格"。**

| 操作 | 空模型 | LLM | 说明 |
|------|--------|-----|------|
| 读取本地文件/传感器 | ✓ | ✗ | LLM无本地文件系统概念 |
| 查询元事实库 | ✓ | ✗ | LLM不知道元事实库存在 |
| 写入本地/执行命令 | ✓ | ✗ | LLM零写入、零执行权限 |
| 请求额外数据 | ✗ | ✗ | 单次请求-响应，LLM无请求资格 |
| 使用外部知识 | ✗ | ✗ | 空模型已筛选，LLM接收不到外部知识 |
| 修改执行计划 | ✓ | ✗ | LLM只返回文本，不返回可执行结构 |
| 返回外部链接 | ✗ | ✗ | 防止钓鱼/恶意链接 |
| 声称"我认为/我知道" | ✗ | ✗ | 协议层禁止，LLM只能"根据您的事实" |

### 7.2 防火墙的本质

```
旧思维（nmp-1.0）：
  LLM是危险的野兽 → 给它戴手铐 → 防止它作恶

新思维（nmp-2.0）：
  LLM没有自己的事实 → 它只活在空模型画的领地里 → 
  领地外的东西它看不到、想不到、做不到 → 无需手铐，因为无手
```

## 八、安全机制

1. **TLS 1.3**：所有通信加密
2. **请求签名**：空模型请求带 HMAC-SHA256 签名，防篡改
3. **响应校验**：LLM响应带校验和，防中间人
4. **超时回退**：LLM响应超时（默认30秒），空模型回退到默认值或拒绝服务
5. **速率限制**：防止LLM被滥用
6. **事实溯源**：每条事实可追溯到元事实库版本和确认者
7. **判例库隔离**：反馈数据本地存储，不上传云端，保护使用者隐私



## 九、版本演进

| 版本 | 特性 |
|------|------|
| nmp-1.0 | 基础事实+策略交换，防火墙式限制 |
| **nmp-2.0** | **事实主权+代表立场+闭环反馈+无资格原则** |
| nmp-2.1 | 支持多轮协商（LLM请求澄清特定事实，空模型响应查询） |
| nmp-3.0 | 支持跨使用者事实授权（企业授权员工访问特定事实子集） |

## 六、结论

### 6.1 总体结论

三份新文档（元事实库、双相理论、云端协议）**严格符合**基准报告确立的所有核心命题，**三份文档之间完全自洽**。

基准报告中的架构级偏差已全部修正，建议项已实施。新文档在保持逻辑一致性的基础上，增强了：
- 事实主权原则（owner/confirmed_by）
- 代表原则（LLM无资格代表自己）
- 自学习闭环（feedback_bounce + 四机制工程化）
- 版本管理（静态快照的时间刻度）
