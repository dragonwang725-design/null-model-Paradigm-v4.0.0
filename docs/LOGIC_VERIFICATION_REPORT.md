## 二、核心命题验证

### 命题1：空模型 = 纯粹判断容器，无记忆、无欲望、无目的

| 子命题 | 基准报告 | 新文档 | 一致性 |
|--------|---------|--------|--------|
| 纯粹判断能力（形式推理、关联、抽象、决策） | `causal_skeleton.py` BFS图距离 | 元事实库："先验计算结构：目的→事实类型映射" | ✅ |
| 无固化内容（不含领域知识、经验、偏见） | 领域知识外置到元事实库 | 元事实库："空模型只读查询，任务后约束丢弃" | ✅ |
| 无目的意识（无目标、欲望、价值判断） | `purpose_parser.py` 只解析结构 | 双相理论："空模型无目标、无欲望、无价值判断" | ✅ |
| 先验计算能力保留（L0） | BFS、约束满足、校验和验证 | 元事实库："先验计算结构"；云端协议："先验计算骨架" | ✅ |

**结论**: 完全一致。新文档将"先验计算能力"明确为"目的→事实类型映射"的结构，不含内容，与基准报告L0/L1分层完全对应。

### 命题2：事实因人而异因目的而异，空模型基于目的提取事实

| 子命题 | 基准报告 | 新文档 | 一致性 |
|--------|---------|--------|--------|
| 没有绝对的事实，事实取决于角度和目的 | `purpose_parser` + `relevance_engine` | 元事实库："没有无目的的事实"；"事实只在特定使用者的特定目的下被提取" | ✅ |
| 空模型以目的为基准提取事实 | `perception_scheduler` 提取前调度 | 元事实库TDA流程图：目的→空模型→查询元事实库 | ✅ |
| 无关目的的信息自动过滤 | `causal_skeleton` 图距离过滤 | 元事实库："空模型按需提取，不是全量加载" | ✅ |
| 元事实库约束提取动作 | 查询元事实库获取允许/禁止列表 | 元事实库："约束层（Constraints）是使用者确认的提取限制" | ✅ |

**结论**: 完全一致。新文档强化了"目的中心制"，明确元事实库不按领域分类、按目的组织。

### 命题3：LLM只负责策略判断，空模型负责最终判断

| 子命题 | 基准报告 | 新文档 | 一致性 |
|--------|---------|--------|--------|
| 空模型提取事实，输入LLM做策略判断 | `llm_client.py` 接收脱敏事实 | 云端协议："空模型向LLM发送事实约束包" | ✅ |
| LLM返回策略方案 | `llm_client.py` 返回策略 | 云端协议："LLM返回立场响应（stance_response）" | ✅ |
| 空模型最终判断：是否符合事实和目的 | `final_judgment.py` 校验约束 | 元事实库TDA流程：裁决AI→输出→事实反馈 | ✅ |
| LLM被笼养，零本地执行权 | **输入边界锁机制** | 云端协议："**无资格原则**——LLM不是被禁止，是根本看不到" | ✅ |
| LLM不知道元事实库存在 | 架构隔离 | 元事实库："LLM完全不可见"；云端协议："LLM不知道元事实库存在" | ✅ |

**结论**: 完全一致。新文档将"笼养"升级为"无资格原则"，本质相同但表述更精确。云端协议nmp-2.0新增`stance`结构（representing/based_on_facts），强化了LLM"代表使用者"而非"自主判断"的定位。

### 命题4：元事实库 = 人类元规则层，决定提取什么

| 子命题 | 基准报告 | 新文档 | 一致性 |
|--------|---------|--------|--------|
| 元事实库是人类的元规则层 | `meta_fact_vault.py` Rule/Knowledge/Log/MetaFact | 元事实库："使用者确认的静态事实集合"；"约束层（Constraints）是使用者确认的事实" | ✅ |
| 包含知识、规则、日志、元事实 | 四类数据结构 + 领域本体映射表 | 元事实库：facts + constraints + ontology_mappings + meta_facts + logs | ✅ |
| 空模型在提取前必须查询元规则 | `perception_scheduler.schedule()` | 元事实库："空模型每次任务开始时查询" | ✅ |
| 隐私不是提取后删除，而是提取动作本身被约束 | **提取前调度** | 元事实库："提取动作本身被元规则约束" | ✅ |
| 元事实库对LLM完全不可见 | 架构防火墙 | 元事实库："LLM完全不可见"；云端协议："LLM不知道元事实库存在" | ✅ |

**结论**: 完全一致。新文档修正了基准报告中的"术语体系不统一"问题（隐私资料库→元事实库），并明确Rules/Constraints是"使用者确认的事实"而非系统预设。



### 命题5：六步流水线是跨本地-云端的完整流程

| 步骤 | 基准报告 | 新文档 | 一致性 |
|------|---------|--------|--------|
| 1. Problem | 用户输入（本地） | 元事实库TDA流程：使用者（目的） | ✅ |
| 2a. Null Model (Meta-Fact Query) | `meta_fact_vault.query_rules()`（本地） | 元事实库：空模型查询元事实库 | ✅ |
| 2b. Null Model (Strategy) | `physical_budget` + `perception_scheduler` + `constraint_locking`（本地） | 元事实库：空模型提取+结构化 | ✅ |
| 3. Full Model (Cognition) | `llm_client.query()`（云端） | 云端协议：LLM接收事实约束包 | ✅ |
| 4. Database (Knowledge) | `meta_fact_vault.query_rules()`（本地） | 元事实库：二次查询（如需要） | ✅ |
| 5. Full Model (Synthesis) | `llm_client.query()`（云端） | 云端协议：LLM生成立场响应 | ✅ |
| 6. Null Model (Judgment/Execution) | `final_judgment.judge()` + `atomic_executor.execute_batch()`（本地） | 元事实库TDA流程：裁决AI→输出→执行 | ✅ |

**增强项（非矛盾）**:
- 新文档在六步之后增加了**闭环反馈**：事实反馈→判例库→人工审计→元事实库更新。这是基准报告建议的"判例库驱动自学习"的工程化实现。

**结论**: 完全一致，且增强了闭环反馈。

### 4.2 新增概念（非偏差，是增强）

|---------|---------|----------------|

### 4.3 潜在细微张力（非逻辑矛盾）

## 六、结论

### 6.1 总体结论

三份新文档（元事实库、双相理论、云端协议）**严格符合**基准报告确立的所有核心命题，**三份文档之间完全自洽**。

基准报告中的架构级偏差已全部修正，建议项已实施。新文档在保持逻辑一致性的基础上，增强了：
- 事实主权原则（owner/confirmed_by）
- 代表原则（LLM无资格代表自己）
- 自学习闭环（feedback_bounce + 四机制工程化）
- 版本管理（静态快照的时间刻度）
